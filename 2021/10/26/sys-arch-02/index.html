软考知识点- 系统开发基础

今天是2021年10月27日，还有9天就软考了，现在开始刷选择题，系统开发基础

系统开发基础

变更策略

  1. 所以需求变更必须遵循变更控制过程
  2. 对于未获得批准的变更，不应该做设计和实现工作
  3. 变更应该由项目变更控制委员会决定实现哪些变更
  4. 项目风险承担者应该能够了解变更数据库的内容
  5. 决不能从数据库中删除或修改改变请求的原始文档
  6. 每一个集成的需求变更必须能跟踪到一个经核准的变更请求

软件需求工程

需求应当分优先等级，不能把所有需求都视为同等重要

需求管理是一个对系统需求及其变更进行了解和控制的过程

需求管理的目标是为软件建立一个基线

提供给软件工程和管理使用：软件计划、产品和活动与软件需求保持一致

变更委员会

负责决定哪些已建议需求变更或新产品特性付诸应用，决定在哪些版本中纠正哪些错误。

CMM

过程域

  1. 需求管理
  2. 软件项目计划
  3. 软件项目跟踪和监督
  4. 软件分包合同管理
  5. 软件质量保证
  6. 软件配置管理

4+1 视图模型

  • 逻辑视图
  • 实现视图
  • 进程视图
  • 部署视图
  • 用例视图

  • 最终用户侧重逻辑视图

  • 系统工程师侧重部署视图

需求开发

确定软件所期望的用户类型

获取各种用户类型的需求

了解实际的用户任务和目标

以及这些任务所支持的业务需求

基于UML需求分析

  1. 利用用例及用例图表示需求。
  2. 利用包图和类图表示目标软件系统的总体框架结构

需求开发与需求管理是相辅相成

需求开发是主线、目标、需求管理是支持、保障

结构化分析法

基本思想是自顶向下，逐层分解，把一个大问题分解成若干个小问题，每个小问题再分解成若干小问题

项目管理

草稿、正式发布和正在修改

系统管理工具

特征

  1. 覆盖整个软件生存周期
  2. 为项目高度提供多种有效手段
  3. 利用估算模型对费用和工作量进行估算
  4. 支持多个项目和子项目的管理
  5. 确定关键路径 ，松弛时间，超前时间和滞后时间
  6. 对项目组成员和项目任务之间的通信给予辅助
  7. 自动进行资源平衡
  8. 跟踪资源的使用
  9. 生成固定格式的报表和剪裁项目报告

成本 估算工具就是一种典型的项目管理工具

项目范围说明书

范围定义

  1. 项目章程
  2. 项目范围管理计划
  3. 组织过程资产
  4. 需求文件

项目时间管理

包括使项目按时完成所必需的管理过程

项目过程：

  1. 活动定义
  2. 活动排序
  3. 活动的资源估算
  4. 活动历时估算
  5. 制定进度计划
  6. 进度控制

项目时间管理

包括使项目按时完成所必须的管理过程。

过程：

  1. 活动定义
  2. 活动排序
  3. 活动的资源估算
  4. 活动历时估算
  5. 制定进度计划
  6. 进度控制

逆向工程

抽象层

  1. 实现层：包括程序的抽象语法树、符号表等信息
  2. 结构级：包括反映程序分量之间相互依赖关系的信息，例如调用图、结构图等
  3. 功能级：包括反映程序段功能及程序段之间的关系的信息
  4. 领域级：包括反映程序分量或程序与应用领域概念之间对应关系的信息

软件测试

软件测试是为了发现错误而执行程序的过程

集成测试

它将已通过单元测试的模块集成在一起，主要测试模块之间的协作性

从组装策略而言，可以分为一次性组装测试和增量式组装两种（包括自顶向下自底向上及混合式）。

集成测试计划通常是在软件概要设计阶段完成的，集成测试一般采用黑盒测试方法

黑盒测试

也称为功能测试

  • 根据规格说明所规定的功能来设计测试用例
    • 不考虑程序的内部结构和处理过程
    • 常用 技术
      • 等价类划分
      • 边值分析
      • 错误猜测
      • 因果图

系统测试

介绍

系统测试的主要内容包括功能测试、健壮性测试、性能测试、用户界面测试、安全性测试、安装与反安装测试

  1. 恢复测试：
    • 恢复测试监测系统的容错能力
  2. 安全性测试：
    • 系统的安全性测试是检测系统的安全机制、保密措施是否完善，主要是为了检验系统的防范能力
  3. 强度测试
    • 对系统在异常情况下的承受能力的测试，是检查系统在极限状态下运行时，性能下降的幅度是否在允许的范围内
  4. 性能测试
    • 检查系统是否满足系统设计方案说明书对性能的要求。
    • 性能测试覆盖了软件测试的各阶段

性能测试

其中性能测试包括负载测试、压力测试、可靠性测试和并发测试

内部确认测试

内部确认测试主要由软件开发组织内部按照软件需求规格说明书进行测试

α测试和β测试

α测试指用户在开发环境下进行测试，通过alpha测试以后的产品通常称为alpha版

β测试是指由用户在实际使用环境下进行测试 通过 beta测试的产品通常称为beta版本

一般在通过beta测试后，才能把产品发布或交付给用户

验收测试

验收测试是指针对软件需求规格说明书，在将会前以用户为主进行的测试。

验收测试的结论是用户确定是否接收该的主要依据

性能

用户必须依靠评价程序来评价机器的性能。

把应用程序用的最多、最频繁的那部分核心程序作为评价计算机性能的标准程序

WEB服务器性能

  • 请求响应时间
  • 事务响应时间
  • 并发用户数
  • 吞吐量
  • 资源利用率
  • 每秒钏系统能够处理的交易
  • 事务的数量

软件开发环境

Software Development Environment SDE

由软件工具集和环境集成

  • 软件工具集
    • 支持软件开发的相关过程、活动和任务年
  • 环境集成
    • 为工具集成和软件开发、维护和管理提供统一的支持，它通常包括数据集成 、控制集成和界面集成

数据集成机制提供了存储或访问环境信息库的统一的数据接口规范；

界面集成机制采用统一的界面形式，提供统一的操作方式；控制集成机制支持各开发活动之间的通信、切换、高度和协同工作。

  • 自检位用于对主关键字进行基于检验位的检查
  • 域检查用于验证数据是否位于合法的取值范围
  • 格式检查按照已知的数据格式对照检查输入数据的格式

环境信息库

是开发环境的核心，用以存储与系统开发有关的信息，并支持信息的交流与共享

过程控制与消息服务器

是实现过程集成和控制集成的基础 。

过程集成时按照具体软件开发过程的要求进行工具的选择与组合

环境用户界面

包括环境总界面和由它实行统一控制的各环境部件及工具的界面。

统一的、具有一致性的用户界面是软件开发环境的重要特征

系统输入

通过内部控制的方式验证输入的数据的有效性

软件工程

RUP软件开发生命周期

一个二维的软件开发模型，强调迭代和增量

好处：在软件开发的早期就可以对关键的影响大的风险进行处理

工作流:

  • 业务建模、需求、分析与设计 、实现、测试部署、配置与变更管理、项目管理以及环境

四个阶段：

初始阶段

定义最终产品视力和业务模型，并确定系统范围

细化阶段

设计及确定系统的体系结构，制定工作计划及资源要求

构造阶段

构造产品并继续演进需求、体系结构、计划直到产品提交

移交阶段

把产品提交给用户使用

面向对象开发方法

基本任务

把面向对象分析模型转换为面向对象的设计模型.

面向对象的分析模型主要由顶层架构图、用命与用例图、领域概念模型构成。

三种类型

实体类、边界类和控制类

实体类

映射需求中的每个实体

控制类

用于控制用例工作的类，一般是由动宾结构的短语转化而来的名词。

控制类用于对一个几个用命所特有的控制行为进行建模，控制对象（控制类的实例）通常控制其他对象。因此它们的行为具有协调性。

边界类

用于封闭在用例内、外流动的信息或数据液压。

边界类是一种用于对系统外部环境与其内部动作之间的交互进行建模的类，用于实现目标软件系统与外部系统或外部设备之间的信息交流和互操作

开发模型

螺旋模型

一种深化软件开发过程模型

它在快速模型的基础上扩展而成，螺旋模型最大的特点在于引入了其他模型不具备的风险分析，以减少损失

软件维护

四种类型

正确性维护

改正在系统开发阶段已发生而系统测试阶段尚未发现的错误

适应必维护

在使用过程中，外部环境（新的硬、软件配置）、数据环境（数据库、数据格式、数据输入/输出方式、数据存储介质 ） 可能 发生变化。

为使软件适应这种变化，而去修改软件的过程就称为适应性维护

完善性维护

在软件的使用过程中，用户往往会提出新的功能与性能要求。为了满足这些要求，需要修改或再开发软件，以扩充软件功能、增强软件性能、改进加工效率、提高软件的可维护性

预防性维护

适应未来软件 硬件环境的变化，应主动增加预防性的新的功能，以使应用系统适应各类变化而不被淘汰

软件系统工具

软件控制工具属于软件维护工具

软件评价工具属于软件管理软件支持工具

设计模式

Bridge模式

  1. 不希望在抽象以及抽象的实现部分之间有一个固定的绑定关系。
  2. 类的抽象以及它的实现都应该可以通过生成子类的方法加以扩充，使用Bridge模式可以对不同的抽象接口和实现部分进行组合，并分别对它们进行扩充
  3. 对一个抽象的实现部分的修改应该对用户不产生影响，即客户的代码不必重新编译。

野生知识

CMM（能力成熟度模型）

涵盖了规划、软件工程、管理、软件开发及维护等技巧

  • 若能确实遵守，可协助提升软件部门的软件设计能力，达到成本、时程、功能与品质的目标

构成产品配置

软件过程模型

软件活动

  1. 软件描述：必须定义软件功能以及使用的限制
  2. 软件开发：也就是软件的设计和实现，软件工程人员制作出能满足描述的软件
  3. 软件有效性验证：软件必须经过严格的验证，以保证能够满足客户的需求
  4. 软件进化。软件随着客户需求的变化不断地改进

瀑布模型

因果关系紧密相连，前一个阶段工作的结果是后一个阶段工作的输入。或者说，每一个阶段都是建筑在前一个阶段的正确结果之上，前一个阶段的错漏会隐蔽地带到后一个阶段。

这种错误有时甚至可能是灾难性的，因此生一个阶段工作完成后，都要进行审查和确认。

面向对象分析模型

主要由顶层架构图、用例图、领域概念模型构成

设计模型则包含以包图表示的软件体系结构图、以交互图表示的用例实现图、完整精确的类图、针对复杂对象的状态图和用以描述流程化处理过程的活动图

敏捷开发

主要开发方法包括

极限编程（XP）/ 自适应软件开发 / 水晶法 / 特性驱动开发

极限编程

遵循四五十二

四大价值观：沟通、简单、反馈、勇气

五个原则 ：快速反馈、简单性假设、逐步修改、提倡更改、优质工作

十二最佳实践：计划游戏、小型发布、隐喻、简单设计、测试先行、重构、结对编程、集体代码所有制、持续集成 、每周工作40小时
